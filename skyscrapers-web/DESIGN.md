# Skyscrapers Web Frontend — Design Document

## Purpose & Scope

### MVP

A browser-based interactive player for Skyscrapers puzzles. Puzzles are loaded from URL parameters (pre-generated by the Rust CLI). The player can fill in cells, receive real-time validation feedback, and detect completion.

### Future (WASM Integration)

Compile `skyscrapers-core`, `skyscrapers-generator`, and `skyscrapers-solver` to WebAssembly via `wasm-pack`. This enables in-browser puzzle generation without a server. The encoding format is designed to make this integration seamless.

## Puzzle Encoding Format

Puzzles are encoded as a compact digit string, passed via the URL query parameter `?p=<encoded>`.

### Structure

```
<n><top[0..n]><bottom[0..n]><left[0..n]><right[0..n]><board row-major n*n>
```

| Segment               | Length | Description                                      |
|-----------------------|--------|--------------------------------------------------|
| `n`                   | 1      | Board size as a single digit (5–9)               |
| `top[0..n]`          | n      | Top clues, left-to-right. `0` = no clue          |
| `bottom[0..n]`       | n      | Bottom clues, left-to-right                      |
| `left[0..n]`         | n      | Left clues, top-to-bottom                        |
| `right[0..n]`        | n      | Right clues, top-to-bottom                       |
| `board[0..n*n]`      | n×n    | Board cells in row-major order. `0` = empty cell |

**Total length**: `1 + 4n + n²`

| n | Total length |
|---|-------------|
| 5 | 46          |
| 7 | 78          |
| 8 | 97          |

### Clue Ordering

The clue order matches the Rust `Clues` struct: **top, bottom, left, right**.

### Example

For an n=5 puzzle:

```
500300010030000003040200000000000000000000000020000
^     ^     ^     ^     ^
n  top[5] bot[5] lft[5] rgt[5]  board[25]
```

### Validation Rules

- Total string length must equal `1 + 4n + n²`
- All characters must be ASCII digits `0`–`9`
- `n` must be in range `5`–`9`
- Clue values must be in `0`–`n` (0 means no clue)
- Cell values must be in `0`–`n` (0 means empty)

## TypeScript Type Definitions

```typescript
// types.ts

type ClueValue = number | null;  // null = no clue

interface Puzzle {
  n: number;                     // 5–9
  board: BoardCell[][];          // n × n, row-major
  clues: {
    top: ClueValue[];            // length n
    bottom: ClueValue[];
    left: ClueValue[];
    right: ClueValue[];
  };
}

interface BoardCell {
  value: number | null;          // 1..n or null (empty)
  given: boolean;                // true = from puzzle definition (not editable)
}

interface GameState {
  puzzle: Puzzle;                       // original puzzle (immutable)
  board: BoardCell[][];                 // player's working board
  selectedCell: [number, number] | null; // [row, col] or null
  errors: Set<string>;                  // set of "r,c" keys with violations
  completed: boolean;                   // all cells filled, no errors
}

type GameAction =
  | { type: "SELECT_CELL"; row: number; col: number }
  | { type: "SET_VALUE"; value: number }
  | { type: "CLEAR_CELL" }
  | { type: "RESET" }
  | { type: "CHECK" };
```

### Encoding Module

```typescript
// encoding.ts

function encodePuzzle(puzzle: Puzzle): string;
function decodePuzzle(encoded: string): Puzzle;  // throws on invalid input
```

## Component Design

### App

- Reads `?p=` URL parameter; decodes puzzle if present
- Otherwise displays a menu of sample puzzles (from `samples.ts`)
- Renders `PuzzlePage` with the selected puzzle

### PuzzlePage

- Owns `GameState` via `useReducer`
- Dispatches actions: `SELECT_CELL`, `SET_VALUE`, `CLEAR_CELL`, `RESET`, `CHECK`
- Runs validation on every board change
- Detects completion: all cells filled and no errors
- **Keyboard handling**:
  - Digits `1`–`n`: set value in selected cell
  - `0` / `Backspace` / `Delete`: clear cell
  - Arrow keys: move selection (clamp at edges)
  - `Tab`: move to next empty cell
  - `Escape`: deselect

### PuzzleGrid

- Pure rendering component
- CSS Grid layout: `(n+2) × (n+2)` to accommodate clues on all four edges
- `grid-template-columns: auto repeat(n, 1fr) auto`
- Corner cells: empty
- Edge cells: `ClueCell`
- Interior cells: `BoardCell`

### ClueCell

- Displays clue value or empty string
- Distinct styling: no border, muted color
- Non-interactive

### BoardCell

- Props: `value`, `given`, `selected`, `hasError`, `onClick`
- **Given cells**: bold font, light gray background, not clickable
- **User cells**: white background, clickable
- **Selected cell**: blue outline or background tint
- **Error cells**: light red background

### NumberPad

- Buttons `1` through `n`, plus a clear button
- Fires `onNumberSelect(value)` or `onClear()`
- Optimized for mobile touch targets
- Highlights the button matching the current cell's value

### GameControls

- **Reset**: clears all user input
- **Check**: triggers validation display
- **New Puzzle**: returns to puzzle selection

## Styling

- CSS Grid for the board layout (clues + board cells)
- Responsive cell size: `min(60px, calc((100vw - 80px) / (n + 2)))`
- Thicker borders on the outer board edge; 1px borders between cells
- Completion state: green flash or border color change
- `tabIndex={0}` on the grid container for keyboard focus
- Mobile: `NumberPad` compensates for small cell sizes at n=8/9

## Validation Strategy

### MVP (Row/Column Uniqueness Only)

```typescript
// validation.ts

function validateBoard(n: number, board: BoardCell[][]): Set<string>;
```

- **Row uniqueness**: if the same non-null value appears twice in a row, both cells are marked as errors
- **Column uniqueness**: same check for columns
- **Range check**: any value outside `1..n` is an error
- Returns `Set<string>` of `"r,c"` keys identifying cells with violations
- Runs on every cell change for immediate feedback

### Future

- **Clue visibility validation**: compute visible building counts and compare against clues
- Can be added once the base "solve feel" is validated

## Implementation Steps

1. **Project scaffolding**: `npm create vite@latest` with `react-swc-ts` template; update root `.gitignore` with `skyscrapers-web/node_modules/` and `skyscrapers-web/dist/`
2. **Types and encoding**: implement `types.ts` and `encoding.ts` with unit tests
3. **Sample puzzles**: generate 3–5 puzzles with the CLI (one n=5, two n=7, one n=8); encode to digit strings and store in `samples.ts`
4. **Grid rendering**: implement `PuzzleGrid`, `ClueCell`, `BoardCell` components
5. **Game state and interaction**: implement `PuzzlePage` with `useReducer`, keyboard handling, cell selection
6. **Validation**: implement `validation.ts` and wire into state updates
7. **Controls and polish**: add `NumberPad`, `GameControls`, completion detection
8. **Styling polish**: responsive sizing, mobile-friendly touch targets, completion animation

## Future WASM Integration Path

1. Create a `skyscrapers-wasm/` crate that wraps `skyscrapers-core` + `skyscrapers-generator` + `skyscrapers-solver`
2. Compile with `wasm-pack` targeting `web`
3. Expose a single function: `generate(n: usize, seed: u64) -> String`
   - Returns the encoded digit string directly (same format as the `?p=` parameter)
   - No need for `Puzzle` Display/FromStr parsing at the WASM boundary
4. Web frontend calls WASM `generate()`, then decodes the result with the existing `decodePuzzle()` function
5. The encoding format is intentionally designed to make this boundary trivial
